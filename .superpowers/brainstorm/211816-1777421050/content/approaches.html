Echo — 3 Architecture Approaches

All use HTTP/SSE MCP transport embedded in mir serve

A

Inline handler in Cockpit

MCP endpoint registered directly in cockpit_srv alongside existing API routes. Echo shares the Cockpit HTTP server and its middleware stack.

Pros

  • Minimal new code
  • Reuses CORS, logging, metrics
  • Single port for UI + MCP

Cons

  • Cockpit grows beyond its responsibility
  • Hard to disable Echo independently
  • Tangled concerns
B ⭐ Recommended

Separate echo_srv package — same port

New internal/servers/echo_srv/ package, same pattern as cockpit_srv. Routes registered on the shared HTTP mux under /mcp/.... Own tool registry, doc embedder, NATS client.

Pros

  • Mirrors existing cockpit_srv pattern exactly
  • Clean boundaries, testable in isolation
  • Single port, separate concerns
  • Can be toggled via config

Cons

  • Slightly more boilerplate than A
C

Separate HTTP server — own port

Echo runs on its own port (e.g. :8484), independent lifecycle from Cockpit. Separate goroutine, separate mux, separate config block.

Pros

  • True isolation
  • Independent enable/disable/TLS

Cons

  • Extra port to configure & expose
  • More infra complexity (k8s, firewall)
  • Overkill for v1
