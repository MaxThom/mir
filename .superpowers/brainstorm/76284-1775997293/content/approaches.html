Implementation Approaches

Three ways to build the welcome page — pick the one that fits

A

Minimal — Extend existing DashboardGrid with readonly

Add a readonly prop to the existing DashboardGrid component that disables edit mode, drag handles, and the add-widget button. The welcome +page.svelte fetches system/welcome from the API, passes it directly to DashboardGrid readonly. No new toolbar — just a static title. The dashboard store is not involved at all (direct API call in the page). Backend seeds system/welcome on startup if absent.

Pros

  • Minimal new code — ~2 files touched
  • All widget types work automatically
  • Grid positions from DB respected

Cons

  • DashboardGrid becomes dual-purpose
  • GridStack overhead for a static page
B ⭐

Dedicated welcome store + simplified page (recommended)

A small welcomeStore fetches system/welcome from the API and holds the dashboard. The +page.svelte (new route /) uses DashboardGrid with readonly=true and a minimal welcome toolbar (title + refresh button only — no tabs, no edit, no add widget). Move existing dashboard to /dashboards/+page.svelte. Clean separation: welcome store is read-only by contract, dashboard store is unchanged. Backend seeds system/welcome on startup.

Pros

  • Clean separation of concerns
  • Existing dashboard store untouched
  • Explicit read-only contract in the store
  • Easy to add welcome-specific logic later

Cons

  • One new store file
  • DashboardGrid still needs readonly prop
C

Custom CSS grid — no GridStack

A welcome-specific grid that renders widgets using plain CSS grid instead of GridStack. Same widget components (WidgetTelemetry, WidgetEvents, etc.) but positioned via CSS grid areas defined in the DB config. No drag/drop library loaded at all on the welcome page.

Pros

  • No GridStack on a page that never needs drag/drop
  • Lighter page weight

Cons

  • Must implement custom grid layout from widget x/y/w/h
  • CSS grid areas need translation from widget coords
  • More code, more maintenance
